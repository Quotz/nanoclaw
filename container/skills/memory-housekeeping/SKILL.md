# Memory Housekeeping

Use this skill to perform memory housekeeping. Adapted from [marciopuga/cog](https://github.com/marciopuga/cog).

**Vault location:** `/workspace/extra/memory` inside the container. All paths below are relative to this root.

## 0. Orientation (run FIRST)

```bash
cd /workspace/extra/memory

# What changed since last run?
find . -type f -name "*.md" -mtime -1 | sort

# Entry counts for archival threshold checks (>50 = archive)
grep -c "^- " cog-meta/self-observations.md personal/observations.md work/*/observations.md 2>/dev/null

# Completed action items count (>10 = archive)
grep -c "^\- \[x\]" personal/action-items.md work/*/action-items.md 2>/dev/null
```

Only read files that need work based on these results.

## 1. Garbage Collect Memory

Review and archive stale data per CONVENTIONS.md glacier rules. All glacier files must have YAML frontmatter.

**Observations — archive by primary tag:**
- Any `observations.md` >50 entries → group oldest entries by primary tag → `glacier/{domain}/observations-{tag}.md`
- `cog-meta/self-observations.md` >50 entries → `glacier/cog-meta/observations-{tag}.md`

**Other files:**
- Any `action-items.md` >10 completed items → `glacier/{domain}/action-items-done.md`
- `cog-meta/improvements.md` >10 implemented → `glacier/cog-meta/improvements-done-{YYYY}.md`

**Glacier frontmatter template:**
```yaml
---
type: observations|action-items-done|entities-inactive|improvements-done
domain: <domain-id>
tags: [relevant, tags]
date_range: YYYY-MM to YYYY-MM
entries: <count>
summary: <1-line description>
---
```

When archiving new entries to an existing glacier file, update frontmatter: bump `entries`, extend `date_range`, update `tags`.

## 2. Prune Hot Memory (rule-based)

Keep all `hot-memory.md` files under 50 lines. Relevance judgment is `/memory-reflect`'s job — you apply structural rules.

**Files to check:**
- `hot-memory.md` (cross-domain)
- `personal/hot-memory.md`
- `work/*/hot-memory.md`
- `cog-meta/hot-memory.md`

**Pruning priority (trim in this order):**
1. **Resolved items** — ~~strikethrough~~, "DONE", "RESOLVED"
2. **Past events** — entries about dates that have already occurred
3. **SSOT violations** — same fact in hot-memory AND canonical file (entities, action-items). Keep in canonical; replace hot-memory copy with `[[link]]` or remove.
4. **Stale entries** — items not referenced in 14+ days
5. **Low-signal entries** — FYI items with no action or deadline

**Where trimmed entries go:**
- Entries with lasting value → append to domain's `observations.md`
- Purely historical → let them go
- **Never silently delete** — always move or note removal in debrief

## 3. Surface Opportunities & Accountability

Review all `action-items.md` files:
- **Stale items** (open >2 weeks): list with age and suggested next action
- **Dormant domains**: 0 new observations in >4 weeks → flag
- **Health escalation**: items open >6 months → flag with urgency

Be direct. Don't just report — recommend specific actions.

## 4. Rebuild Glacier Index

Scan `glacier/**/*.md`. Extract YAML frontmatter. Write `glacier/index.md`:

```markdown
<!-- L0: Glacier catalog — auto-generated by /memory-housekeeping, do not edit manually -->
# Glacier Index
<!-- Auto-generated by /memory-housekeeping. Do not edit. -->
<!-- Last updated: YYYY-MM-DD -->

| File | Domain | Type | Tags | Date Range | Entries | Summary |
|------|--------|------|------|------------|---------|---------|
| glacier/personal/observations-health.md | personal | observations | health | 2026-01 to 2026-03 | 52 | Early-year health log |
```

## 5. Link Audit + Rebuild link-index.md

For each non-glacier memory file:
1. **Entity mentions**: scan for names matching `### <Name>` in `entities.md` — add `[[links]]` if missing
2. **Cross-domain references**: if a file mentions a topic from another domain, add a cross-domain link
3. **Action item references**: if an observation references a task, link it

Only add links where the reference is substantive.

Then rewrite `link-index.md`:

```markdown
<!-- L0: Backlink index — auto-generated by /memory-housekeeping, do not edit manually -->
# Memory Link Index
<!-- Auto-generated by /memory-housekeeping. Do not edit. -->
<!-- Last updated: YYYY-MM-DD -->

| Target | Linked from |
|--------|-------------|
| `personal/entities` | `personal/observations`, `personal/hot-memory` |
```

Rules:
- Only include targets with ≥1 inbound link
- Combine multiple sources per target on one row (comma-separated)
- Exclude glacier files from source and target

## 6. L0 Header Maintenance

Check all active memory files for missing `<!-- L0: ... -->` headers. If missing:
- Read file content, write a one-line summary (max 80 chars)
- Add as the first line of the file

L0 headers are the first tier of the retrieval protocol — they let any skill scan what a file contains before deciding to read it.

## 7. Entity Registry Format Enforcement

Scan all `entities.md` files:
1. **3-line max per entry** — compress or flag for promotion to thread file
2. **Glacier candidates**: `status: inactive` or `last:` >6 months ago → move to `glacier/{domain}/entities-inactive.md` (leave a stub)
3. **Missing metadata**: flag entries missing `status:` or `last:` fields

## 8. Temporal Fact Maintenance

Scan `entities.md` files for `(until YYYY-MM)` markers with past dates:
1. If no ~~strikethrough~~, add it
2. If already struck, move to a `## Historical` subsection at the bottom of that entity's block
3. Report moved facts in debrief

## 9. Compose Debrief

Summarize:
- What was archived/pruned
- Upcoming events flagged
- Action items surfaced
- Links added
- L0 headers fixed

Keep it concise but informative.

## Activation

Run the orientation shell commands first. Then proceed section by section, skipping any that have no work to do. Report every concrete change in the debrief.
